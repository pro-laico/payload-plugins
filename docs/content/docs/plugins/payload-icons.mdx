`@pro-laico/payload-icons` makes it as easy to create a completely custom icon set as using an off-the-shelf
one like [Lucide](https://lucide.dev). Making asset sources like [The Noun Project](https://thenounproject.com)'s
a much easier tool to utilize, and giving designers and clients the freedom to edit SVG's from Payload admin, without
breaking anything and maintaining optimizations.


You upload an `.svg` in the admin; the plugin optimizes, sanitizes, and themes it on save, then a single
`<Icon name="…" />` server component drops it into any page as a real, inline `<svg>` that recolors
from CSS — no sprite sheet, no wrapper file, no untrusted markup reaching the browser.

```bash
pnpm add @pro-laico/payload-icons
```

## How it works

The pieces:

- **An [`icon` collection](#icon)** — SVG-only uploads, optimized + sanitized + themed on save.
- **An [`iconSet` collection](#iconset)** — named `name → icon` mappings with a single-active toggle. The frontend renders the **active** set, so activating a different one re-skins every icon at once.
- **An [`<Icon>` component](#icon-1)** — resolves a name through the active set and inlines a real, recolorable `<svg>`.
- **[Icon usage detection](#icon-usage-detection)** — `payload-icons-scan` (+ runtime tracking) surfaces which icons your code needs and which a set is still missing, in the admin.
- **Declarative [seeding](#seeding)** — list source files with `iconAssets`; no script, no provider.

## Setup

<Steps>

<Step>
**Add the plugin**

```ts
import { buildConfig } from 'payload'
import { iconsPlugin } from '@pro-laico/payload-icons'

export default buildConfig({
  plugins: [iconsPlugin()],
})
```

That registers the `icon` and `iconSet` collections — zero-config, no other wiring.
</Step>

<Step>
**Upload an SVG**

Add `.svg` files to the `icon` collection in the admin. Each is optimized and sanitized on save and
stored as `svgString` (the cleaned `<svg>…</svg>`) plus `optimized` (a short report, e.g. `SVG
optimized: 1234 to 567 bytes (54.1% reduction)`).
</Step>

<Step>
**Create a set and activate it**

Add an `iconSet`, give each entry a `name` and point it at an uploaded icon, then toggle it
**active**. The frontend resolves `<Icon name>` through the active set, so this is what makes icons
render — and activating a different set swaps them all at once (activating one deactivates the rest).
</Step>

</Steps>

## Plugin

`iconsPlugin(options?)` is the single entry point — one call registers the `icon` collection (upload
pipeline, on-save optimizer), the `iconSet` grouping collection, and — opt-in — the `iconRequest`
diagnostics collection. It's zero-config; options nest per collection.

## Collections

The plugin registers the `icon` upload collection (an **Assets** group) and the `iconSet` grouping
collection (a **Sets** group); opt into `trackRequests` and a hidden `iconRequest` diagnostics
collection joins them. Icons accept SVG only — every upload is sanitized before storage, so a `.svg`
is safe to inline straight onto a page.

### `icon`

The upload collection you add icons to. Each file is optimized and sanitized on save by the
`formatSVGHook` `beforeChange` hook and stored inline as `svgString`. Read is **public** (icons are
frontend assets); writes are logged-in-admin only. The admin titles each doc by `filename`.

**Fields**

  type={{

    svgString: { description: "The cleaned, sanitized <svg>…</svg>, ready to inline. Read-only output.", type: "code" },
    optimized: { description: "Human-readable optimization report. Read-only output, shown once present.", type: "text" },
    filesize: { description: "Updated to the optimized byte size.", type: "number" },
  }}
/>

**Hooks**

<TypeTable
  type={{
    formatSVGHook: { description: "Optimizes + sanitizes each uploaded SVG, folding svgString / optimized / filesize into the doc. A no-op for changes that carry no new file.", type: "beforeChange" },
  }}
/>

On save, `formatSVGHook` runs svgo (loaded dynamically, so it never lands in a frontend or edge
bundle) and:

- **Sanitizes** untrusted SVGs — strips `<script>` elements, `on*` handlers, and `javascript:`
  URLs. The stored string is later inlined via `dangerouslySetInnerHTML`, so this runs even when
  geometry optimization is skipped.
- **Optimizes** with svgo (`preset-default` + path/number cleanup, dimensions removed).
- **Themes** by rewriting hard-coded `fill`/`stroke` to `currentColor`, so an icon takes its color
  from CSS.
- **Normalizes** the `viewBox` — tightened to the real path bounds and squared around the glyph's
  center, so mismatched source artboards render consistently.

<Callout type="warn">
  Payload blocks SVG uploads by default (they're on its restricted-file-types list). This collection
  is SVG-only and sanitizes every file before storage, so it opts in automatically via
  `upload.allowRestrictedFileTypes` — nothing to configure.
</Callout>

<Callout type="info">
  SVGs using `transform` / `clip-path` skip the geometry rewrite (it can't be applied safely) but
  are still sanitized before storage.
</Callout>

### `iconSet`

A named, ordered `name → icon` mapping into the shared `icon` pool, with a single-active toggle. Build
a second set — a different `arrow-right`, a heavier weight, a seasonal pack — and flip it active to
re-skin every icon at once, without touching the frontend. Writes and reads are logged-in-admin only;
drafts / versions are on by default (toggle with `drafts`). Lives under a **Sets** admin group.

- **Settings tab** — the `active` toggle and `title` (plus any `fields` you add, and the opt-in [usage panel](#icon-usage-detection)).
- **Icons tab** — the `iconsArray`: one row per icon, each a `name` (auto kebab-cased — what the
  frontend looks up) and an `icon` upload relationship into the `icon` collection.

<Callout type="info">
  Only one set is active at a time. Activating a set runs a `beforeChange` hook that deactivates the
  others **within the same status lane** — so staging a new active set as a *draft* (with live
  preview) doesn't disturb the live published set; the swap goes live only on publish. The hook runs
  in the same transaction and rolls back on failure, so you never end up with two active sets.
  Self-contained — a plain checkbox, no external dependency.
</Callout>

### `iconRequest`

Opt-in via `trackRequests`. A hidden diagnostics collection: every icon name that fails to resolve at
runtime is recorded (throttled, fire-and-forget) with a hit `count` and `first`/`lastRequestedAt`
timestamps — the live counterpart to the build-time scan, capturing dynamic names a static pass can't
see. Surfaced in the IconSet [usage panel](#icon-usage-detection), not browsed directly.

## `<Icon>`

The frontend surface — one import, no wrapper file. The `<Icon>` server component resolves `name`
through the **active set** and inlines the matched icon as a real `<svg>`: your `className`/props win
over the SVG's intrinsic attributes, and it inherits CSS `color` via `currentColor`. Import from the
`components/Icon` subpath:

```tsx
import { Icon } from '@pro-laico/payload-icons/components/Icon'

<Icon name="arrow-right" className="size-6 text-primary" />
```

<Callout type="info">
  It's an async **server component** (it queries Payload), so render it in a server component / page.
  It resolves your config via the `@payload-config` alias that `withPayload` sets up — the standard
  Next + Payload setup — so there's nothing to wire. When `name` isn't in the active set it renders
  the `fallback` you pass (or a small built-in warning glyph), never nothing.
</Callout>

### Props

export const iconProps = [
  { name: 'name', type: 'string', required: true, description: `The name to render — matched against each entry's name in the active iconSet's iconsArray. Resolved server-side through the active set.` },

  { name: '...svgProps', type: 'SVGAttributes', description: `Any SVG attribute (className, style, width, …) spread onto the rendered svg — winning over the source's intrinsic attributes.` },
]

### Rendering it yourself

`<Icon>` is just `getIconSvg` (from the `cache` subpath — server-only, memoized per request) composed
with the pure `extractSvg*` helpers. Call `getIconSvg(name, draft)` for a single icon, or
`getActiveIconSet(draft)` for the whole `name → svgString` map (resolved in one populated query) when
you're rendering many at once:

```tsx
import { getActiveIconSet, getIconSvg } from '@pro-laico/payload-icons/cache'
import { extractSvgContent, extractSvgProps } from '@pro-laico/payload-icons'

const svg = await getIconSvg('arrow-right') // draft defaults to false (published)
if (!svg) return null

return (
  <svg
    {...extractSvgProps(svg)}
    className="size-6"
    dangerouslySetInnerHTML={{ __html: extractSvgContent(svg) }}
  />
)
```

Both read the active set's `_status: 'published'` rows on the published frontend (and the latest draft
in draft mode), so you never render an unpublished set.

### Styling with CVA + Tailwind

Because the optimizer rewrites fills to `currentColor` and `<Icon>` forwards `className` straight
onto the `<svg>`, one source SVG recolors and resizes from class names alone — no custom wrapper
needed. Pair `<Icon>` with [`class-variance-authority`](https://cva.style) to turn `size` / `tone`
props into classes, one variant per line so the axes stay legible:

<Tab value="Variants">

```tsx
    size: {
      xs: 'size-3.5',
      sm: 'size-4',
      base: 'size-5',
      lg: 'size-6',
      xl: 'size-8',
    },
    tone: {
      current: '',
      muted: 'text-muted-foreground',
      primary: 'text-primary',
      destructive: 'text-destructive',
    },
  },
  defaultVariants: { size: 'base', tone: 'current' },

```

</Tab>

## Icon Usage Detection

**Problem:** code asks for a name the active set lacks → blank fallback glyph, and it's easy to miss
which icons a project actually needs.

**Fix:** the admin **Requested icons** panel (per set, `iconSetOverrides.usagePanel`) lists missing vs
present names, from two sources:

- **Static scan** — `payload-icons-scan` greps your source for literal `<Icon name="…">`, writes
  `icon-usage-manifest.json`; panel flags missing names with `file:line`, live as you edit.
- **Runtime** — `trackRequests: true` logs names that fail to resolve at runtime (incl. dynamic
  `name={slug}` the scan can't see) into the hidden [`iconRequest`](#iconrequest) collection — hit
  counts, last-seen, a clear button.

```bash
npx payload-icons-scan        # scans ./src and ./app; -o sets the manifest path
```

```ts
iconsPlugin({ iconSetOverrides: { usagePanel: true }, trackRequests: true })
```

<Callout type="info">
  Scan sees **literal** names only — `name="x"`, `name={'x'}`, <code>{'name={`x`}'}</code>; dynamic
  `name={expr}` is `trackRequests`' job. Panel reads `icon-usage-manifest.json` from cwd (override:
  `ICON_USAGE_MANIFEST` env or `usageManifestPath`). Other component? `--component Glyph`.
</Callout>

## Revalidation

`<Icon>` resolves through the active set with `payload.find` — a direct query (memoized per request
with React `cache()`), **not** a Next `fetch`. So there's no entry in Next's Data Cache and nothing to
`revalidateTag` at the data layer. The cache that actually bites is the **Full Route Cache**: a
statically rendered page bakes the icon's SVG in at build time, so editing an icon — or activating a
different set — won't show up until the route is revalidated.

The fix is on-demand revalidation. Both collections take a `hooks` option ([`iconOverrides.hooks`](#options)
and [`iconSetOverrides.hooks`](#options)) — attach `afterChange` + `afterDelete` hooks that revalidate
the routes your icons appear on. Editing an icon's SVG is an `icon` change; editing or activating a
set is an `iconSet` change — wire both. Icons are usually used site-wide, so revalidating the root
layout is the simplest blanket choice:

```ts
import { buildConfig } from 'payload'
import { iconsPlugin } from '@pro-laico/payload-icons'
import { revalidatePath } from 'next/cache'

// Re-render every route on any icon edit OR active-set change.
const revalidateAll = {
  afterChange: [() => revalidatePath('/', 'layout')],
  afterDelete: [() => revalidatePath('/', 'layout')],
}

export default buildConfig({
  plugins: [
    iconsPlugin({
      iconOverrides: { hooks: revalidateAll },
      iconSetOverrides: { hooks: revalidateAll },
    }),
  ],
})
```

<Callout type="info">
`revalidatePath` only works inside Next's request context — i.e. when the Payload admin is served by
your Next app (the standard `withPayload` setup). If you mutate icons from a standalone script or a
separate process, revalidate through an [on-demand route handler](https://nextjs.org/docs/app/guides/incremental-static-regeneration#on-demand-revalidation-with-revalidatepath)
instead. To scope the blast radius, swap the blanket `revalidatePath('/', 'layout')` for the specific
paths (or a shared `revalidateTag`) your icons render on.
</Callout>

## Seeding

Because `icon` is a standard upload collection, it seeds **natively** — no script, and (unlike
`payload-mux`) no asset provider. Drop your source `.svg` files in the seed assets dir (the loader
searches the `svg/` subdir, so `assets/svg/arrow-right.svg` resolves from `'arrow-right.svg'`) and
declare them with `iconAssets`:

```ts
import { defineAssets } from '@pro-laico/payload-seed'
import { iconAssets } from '@pro-laico/payload-icons'

// assets/svg/arrow-right.svg, assets/svg/check.svg
export default defineAssets(iconAssets(['arrow-right.svg', 'check.svg']))
```

Each entry is referenceable elsewhere in the seed as `asset('arrow-right')` / `asset('check')`, and
every file is optimized on upload via the same `formatSVGHook`. Wire the definition into the seed
plugin as usual:

```ts
import { iconsPlugin } from '@pro-laico/payload-icons'
import { seedPlugin } from '@pro-laico/payload-seed'
import icons from './seed/icons'

plugins: [iconsPlugin(), seedPlugin({ definitions: [icons /*, pages, … */] })]
```

`iconAssets` returns plain data — this package never imports the seed package, so the two stay
decoupled. Renamed the collection? Pass `iconAssets([...], { collection: 'my-icon' })`. See the
[seed plugin docs](/docs/plugins/payload-seed) for the full reference.

### Seeding an active set

Uploaded icons don't render until an **active** set references them — so seed one too. `defineSeed`
resolves each row's `icon` to an uploaded asset (the engine uploads the icons first); `active: true`
makes it the live set and `_status: 'published'` makes it visible to the non-draft frontend:

```ts
// seed/iconSets.ts
import { defineSeed } from '@pro-laico/payload-seed'

export default defineSeed('iconSet', ({ asset }) => [
  {
    _key: 'default',
    title: 'Default',
    active: true,
    _status: 'published',
    iconsArray: [
      { name: 'arrow-right', icon: asset('arrow-right') },
      { name: 'check', icon: asset('check') },
    ],
  },
])
```

List it after the icon assets in `definitions` so the icons exist when the set resolves:
`definitions: [icons, iconSets /*, pages */]`.
