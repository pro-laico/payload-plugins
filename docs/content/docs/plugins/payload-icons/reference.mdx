## Plugin options

`iconsPlugin(options?)` is the single entry point. One call registers the `icon` collection (upload
pipeline, on-save optimizer), the `iconSet` grouping collection, and the `iconRequest` diagnostics
collection. It's zero-config: everything is on by default.

Every collection the plugin registers is one key under `collections`, on a single axis — `false`
means "don't register it", an object means "override it". Admin-only UI toggles live under `admin`.

export const options = [
  { name: 'enabled', type: 'boolean', default: 'true', description: `When false, the plugin is a no-op; no collections are registered.` },
  { name: 'collections.icon', type: 'IconCollectionOverrides', default: '{}', description: `Overrides for the icon upload collection. slug / adminGroup / access / upload replace the defaults; fields and hooks are APPENDED after the built-ins (never replacing them), so your beforeChange runs after formatSVGHook and sees the already-optimized SVG. The icon collection is always registered — there is no false.` },
  { name: 'collections.iconSet', type: 'false | IconSetCollectionOverrides', default: '{}', description: `false skips the iconSet collection entirely (only icon is registered) — use it when you want icons in the CMS but not the grouping/active-set concept. An object overrides the collection: slug, live-preview wiring, set-level fields, per-row fields, a drafts toggle. Like collections.icon, fields and hooks are APPENDED after the built-ins (so your beforeChange runs after the single-active hook), not replaced.` },
  { name: 'collections.iconRequest', type: 'false | IconRequestCollectionOverrides', default: '{}', description: `false skips the iconRequest diagnostics collection and its clear endpoint — nothing tracks missing icons. Otherwise <Icon> records every name that fails to resolve at runtime (throttled, fire-and-forget), including dynamic names a static scan can't see, surfaced in the usage panel. An object overrides the collection: additive fields/hooks (it has no built-in hooks). To force-off only the recorder at runtime, leave the collection registered and set ICON_USAGE_TRACKING=false.` },
  { name: 'admin.usagePanel', type: 'boolean', default: 'true', description: `The IconSet "Requested icons" panel: shows which icons your code needs vs what a set provides (scanned live in dev, from the manifest in prod, plus runtime misses). Set false to omit it.` },
]

Pass options to customize. The **Reference** tab is the interactive view; **TypeScript** is the
same shape in code, every defaulted option written out.

```ts
import { iconsPlugin } from '@pro-laico/payload-icons'

// Defaulted options written out. This is what `iconsPlugin()` does with no args.
iconsPlugin({
  enabled: true,
  // one key per registered collection: `false` to skip it, an object to override it
  collections: {
    // the icon upload collection (slug / adminGroup / access / fields / hooks / upload):
    icon: {
      // slug: 'icon',
      // access: { read: () => true },
      // fields: [ /* … */ ],
      // hooks: { /* … */ },
    },
    // the iconSet collection; `false` omits it:
    iconSet: {
      // fields: [{ name: 'description', type: 'textarea' }],
      // iconRowFields: [{ name: 'aliases', type: 'text', hasMany: true }],
    },
    // the iconRequest diagnostics collection; `false` omits it (no runtime miss tracking):
    iconRequest: {},
  },
  admin: {
    usagePanel: true, // the "Requested icons" panel; set false to omit
  },
})
```

## Environment variables

Two env vars tune usage detection at runtime, independent of the plugin options.

<TypeTable
  type={{
    ICON_USAGE_TRACKING: { description: "Set to false to force-off only the runtime miss recorder, without changing the collections.iconRequest option. The iconRequest collection stays registered.", type: "boolean" },
    ICON_USAGE_MANIFEST: { description: "Overrides the production manifest path the Requested icons panel reads. Default: icon-usage-manifest.json in cwd.", type: "string" },
  }}
/>

## CLI commands

| Command | What |
| --- | --- |
| `payload-icons-scan` | Scan source for `<Icon name>` and write the usage manifest (`icon-usage-manifest.json`). Run it in your production build so the **Requested icons** panel works where source isn't on disk. Scans `./src` and `./app`; `-o` sets the manifest path, `--component Glyph` scans a different component. See [Collections](/docs/plugins/payload-icons/collections#icon-use-detection). |

## Endpoints

None public. Icon-set activation happens in the admin, or through the companion
[`@pro-laico/payload-dev-tools`](/docs/plugins/payload-dev-tools) `/dev` panel.

## Exports

| Export | From | What |
| --- | --- | --- |
| `iconsPlugin` | `@pro-laico/payload-icons` | The plugin factory and single entry point. |
| `extractSvgContent` | `@pro-laico/payload-icons` | Pull the inner markup out of an `svgString` to inline it in your own `<svg>`. |
| `extractSvgProps` | `@pro-laico/payload-icons` | Parse an `svgString`'s root attributes (viewBox, etc.) onto your own `<svg>`. |
| `createIcon` | `@pro-laico/payload-icons/components/Icon` | Factory that takes a Payload handle and returns the `<Icon name="…" />` server component — `const Icon = createIcon(getPayload({ config }))` (resolves through the active set). |
| `getIconSvg` | `@pro-laico/payload-icons/cache` | Resolve one icon name to its `svgString` through the active set (server-only, memoized). |
| `payload-icons-scan` | `@pro-laico/payload-icons` | CLI that scans source for `<Icon name>` and writes the usage manifest. |
