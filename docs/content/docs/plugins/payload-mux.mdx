`@pro-laico/payload-mux` brings [Mux Video](https://www.mux.com/) to Payload. It adds a
**Videos** collection (or extends one you name) whose admin field uploads directly to Mux,
exposes virtual playback / poster / gif URLs, and stays in sync with Mux through a webhook —
delete a video in Payload and its Mux asset goes too, and vice versa.

```bash
pnpm add @pro-laico/payload-mux
```

## Setup

Set your Mux credentials as the standard `MUX_*` environment variables — the plugin and the
Mux SDK read them automatically, so adding the plugin takes no arguments. Then point a Mux
webhook at `/api/mux/webhook` (or `<your routes.api>/mux/webhook` if you've customized
Payload's API route).

```bash
MUX_TOKEN_ID=...
MUX_TOKEN_SECRET=...
MUX_WEBHOOK_SECRET=...        # signed playback also: MUX_SIGNING_KEY, MUX_PRIVATE_KEY
```

## The Videos collection

The plugin registers a `mux-video` collection. Mux fills in the read-only metadata; you set
`title` and an optional `posterTimestamp`.

| Field | Type | Notes |
| --- | --- | --- |
| `muxUploader` | ui | The admin uploader + player. |
| `title` | text | Unique, required. |
| `assetId` | text | Mux asset id (read-only). |
| `duration` | number | Read-only. |
| `posterTimestamp` | number | Seconds into the video for the poster; defaults to the middle. |
| `aspectRatio` / `maxWidth` / `maxHeight` | — | Read-only, from Mux. |
| `playbackOptions[]` | array | Per playback id: `playbackId`, `playbackPolicy`, and virtual `playbackUrl` / `posterUrl` / `gifUrl`. |

## Use a video

Relate to `mux-video` from any collection, then play it back however you like — `playbackId`
and the virtual `playbackUrl` (an HLS `.m3u8`) work with any Mux frontend or a custom HLS
player. The example below uses
[`@mux/mux-player-react`](https://www.npmjs.com/package/@mux/mux-player-react) — `pnpm add
@mux/mux-player-react` if you go that route:

```tsx
import config from '@payload-config'
import MuxPlayer from '@mux/mux-player-react'
import { getPayload } from 'payload'

async function Page() {
  const payload = await getPayload({ config })

  const video = await payload.findByID({
    collection: 'mux-video',
    id: 'example',
  })

  return (
    <MuxPlayer
      // Using the playback ID
      playbackId={video.playbackOptions![0].playbackId!}
      // Or use the playback URL
      src={video.playbackOptions![0].playbackUrl!}
      // Poster
      poster={video.playbackOptions![0].posterUrl!}
    />
  )
}

export default Page
```

## Endpoints

The plugin registers three endpoints under Payload's API route:

- `POST /api/mux/upload` — mint a Mux direct-upload (the admin uploader posts files to it).
- `GET /api/mux/upload?id=…` — read a direct-upload, to pick up its `asset_id`.
- `POST /api/mux/webhook` — verify and apply Mux events: set metadata on ready/updated,
  delete on deleted, and (with `autoCreateOnWebhook`) backfill assets uploaded in Mux.

## The webhook: what depends on it

On upload, the `beforeChange` hook polls the new asset for **~6 seconds** and fills in its
playback metadata (`playbackOptions`, `duration`, `aspectRatio`, …) if Mux finishes encoding
in that window. Anything slower is saved with just its `assetId` and left for the webhook.

**Needs the webhook:**

- **Metadata for videos that take longer than ~6s to encode** — the common case for real
  videos. Without the webhook they're saved without `playbackOptions`, so they never become
  playable (the poll already gave up, and re-saving won't re-fetch).
- **Mux → Payload delete sync** — deleting an asset in the Mux dashboard removes the Payload
  doc only via the `video.asset.deleted` event.
- **`autoCreateOnWebhook` backfill** — importing assets you uploaded directly in Mux.

**Does _not_ need the webhook:**

- The upload itself, and metadata for short videos (set synchronously by the 6s poll).
- **Payload → Mux delete** — handled by the collection's `afterDelete` hook, not the webhook.
- **Seeding** (`seedMuxVideos`) — the 6s limit does **not** apply here. The seeder waits for the
  asset to finish encoding (up to ~2 min) and writes the full metadata into the doc itself, so
  seeded videos are playable straight away whatever their length, webhook or not. (A clip that
  takes longer than ~2 min throws rather than creating an incomplete doc.)
- Playback of already-populated videos — the URLs compute on read from the stored playback id.

### Can it work without a live site?

**No — the webhook needs a publicly reachable URL.** Mux *pushes* events to your endpoint, so
its servers have to be able to POST to it. The Mux SDK doesn't change that: it only **verifies
the signature** on the receiving end — it can't make Mux reach `localhost`. So:

- **Production**: set `MUX_WEBHOOK_SECRET` in the deployed environment and point the Mux
  dashboard webhook at `https://your-site/api/mux/webhook`.
- **Local dev**: expose localhost with a tunnel (`cloudflared tunnel --url http://localhost:3000`
  or ngrok), set that URL as the webhook, and keep `MUX_WEBHOOK_SECRET` in your local env.

Without a tunnel or deployment, short-video uploads, Payload→Mux delete, and seeding all still
work — only the webhook-dependent items above wait until the endpoint is reachable.

## Seeding from local files

Local-file seeding lives in the **seed plugin**, not here — the two packages stay fully
decoupled (neither imports the other). [`@pro-laico/payload-seed`](/docs/plugins/payload-seed)
exposes a Mux helper on its `./mux` subpath, `seedMuxVideos`, that pushes local clips into your
Mux account from a script. It reads this plugin's credentials from `config.custom`, so you pass
only `payload` and the videos.

**What it does, per video:** creates a Mux direct upload → PUTs the local file's bytes →
**waits for Mux to finish encoding** (up to ~2 min) → creates the `mux-video` doc with the
ready asset's metadata, tagged (Mux `passthrough`) so a reseed clears only what it made.
Because it waits for `ready`, seeded videos carry full `playbackOptions` and are **immediately
playable — no webhook needed**, whatever their length (a clip needing more than ~2 min throws
rather than creating an incomplete doc).

```ts
// scripts/seed-mux.ts — run with `payload run scripts/seed-mux.ts`
import { seedMuxVideos } from '@pro-laico/payload-seed/mux'
import { getPayload } from 'payload'
import config from '../src/payload.config'

const payload = await getPayload({ config })

await seedMuxVideos(payload, {
  // One entry per clip to upload + create:
  videos: [
    {
      title: 'Intro', // doc title — must be unique in the collection
      file: 'intro.mp4', // local file, relative to `dir` → seed-assets/intro.mp4 (nested paths ok)
      playbackPolicy: 'public', // 'public' (default) or 'signed'
      posterTimestamp: 3, // optional poster frame, seconds in (default: middle of the video)
    },
  ],
  dir: 'seed-assets', // base dir each video's `file` resolves against (default: 'seed-assets')
  clear: 'tagged', // reseed cleanup: 'tagged' = only seed-created assets+docs; 'all' = wipe the Mux env
  // collection: 'mux-video',   // override the collection slug
  // corsOrigin: '*',           // CORS origin for the direct upload
  // initSettings: { ... },     // override Mux creds; omitted fields read their MUX_* env var
})
```

The Mux SDK ships with the seed plugin, so there's nothing extra to install. See the
[seed plugin docs](/docs/plugins/payload-seed#mux-video-seeding) for the full reference.
