`@pro-laico/payload-seed` is an automatic, type-safe helper for writing seed values for your
collections, globals, and more. It rides the same `payload generate:types` you already run, so the
typing stays accurate and up to date on its own — which means you **and** your AI tools get real
feedback while writing seed content, instead of guessing at shapes.

It also ties into the other Pro Laico plugins, so uploading assets — and working with those plugins
in general — just falls into place.

```bash
pnpm add @pro-laico/payload-seed
```

## How it works

The plugin is a handful of standard pieces:

- **Seed files** — one `defineSeed` / `defineGlobalSeed` export per collection or global, typed against your Payload types.
- **Typed tokens** — `ref()` and `asset()` point at other docs and uploads by handle, not raw ids.
- **Assets** — `defineAssets` declares the files to upload; they upload first so `asset()` resolves.
- **The plugin** — `seedPlugin` registers the command, endpoint, optional admin button, and ref-checking types.
- **Run order** — one engine validates, graphs the dependencies, topo-sorts, clears, uploads, and creates.
- **External assets** — a provider seam lets a collection whose bytes live outside Payload (e.g. Mux) seed like an image asset.

## Setup

<Steps>

<Step>
**Add the plugin**

Hand it your `definitions` — every `seed.ts` export plus the `defineAssets` export. The same array
drives the run and the injected types.

```ts
import { buildConfig } from 'payload'
import { seedPlugin } from '@pro-laico/payload-seed'
import services from './collections/Services/seed'
import posts from './collections/Posts/seed'
import assets from './seed/assets'

export default buildConfig({
  plugins: [seedPlugin({ definitions: [assets, services, posts], adminButton: true })],
  // ...
})
```
</Step>

<Step>
**Generate the admin import map** (only with `adminButton: true`)

The admin button is registered by string path, so regenerate the import map when you enable it (skip
this step if you only run the seed via the CLI / endpoint):

```bash
pnpm payload generate:importmap
```
</Step>

<Step>
**Enable, then run**

The seed is gated by the `ENABLE_SEED` kill switch (off by default). Set it, then run the seed one
of three ways — the bundled command, the admin button (`adminButton: true`), or the endpoint:

```bash
ENABLE_SEED=true pnpm payload seed   # CLI (Local API)
# or click "Seed your database" in the admin header
# or POST /api/seed  (authenticated admin user)
```

<Callout type="warn">
  The seed is **destructive** — it clears and recreates the seeded collections (and the asset
  upload collections). `ENABLE_SEED` is off by default: opt in per-environment while iterating,
  and never set it in production.
</Callout>
</Step>

</Steps>

## Plugin

`seedPlugin(options?)` is the single entry point — one call registers the `payload seed` command,
the `POST /api/seed` endpoint, the optional admin button, and the type augmentation that checks your
refs (all detailed below). It does nothing useful without `definitions`.

### Options

export const options = [
  { name: 'definitions', type: 'SeedDefinition[]', description: `The seed definitions — your defineSeed / defineGlobalSeed exports plus the defineAssets export. The same array feeds the seed run and the typed SeedRegistry injected into payload-types.ts. Omit it and the seed runs but warns there is nothing to do.` },
  {
    name: 'assets',
    type: '{ dir?: string; collection?: string }',
    default: "{ dir: 'assets', collection: 'media' }",
    description: `Where source files live and which upload collection they are created in.`,
    options: [
      { name: 'dir', type: 'string', default: "'assets'", description: `Directory holding the source files, relative to the project root.` },
      { name: 'collection', type: 'string', default: "'media'", description: `Default upload collection assets are created in. A per-file spec can override it.` },

  },
  { name: 'assetProviders', type: 'SeedAssetProvider[]', default: '[]', description: `External-asset providers, such as muxAssetProvider() from the Mux plugin. ## Defining seeds

`defineSeed(slug, build)` — one default export per collection. Give each record a `_key` (a local
reference handle), and point at other docs / uploads with the `ref()` / `asset()` tokens (supplied
to the builder) instead of raw ids:

<Tabs items={['Services', 'Posts']}>

<Tab value="Services">

```ts
// collections/Services/seed.ts
import { defineSeed } from '@pro-laico/payload-seed'

export default defineSeed('services', ({ asset }) => [
  { _key: 'consulting', title: 'Consulting', slug: 'consulting', image: asset('serviceA') },
])
```

</Tab>

<Tab value="Posts">

```ts
// collections/Posts/seed.ts
import { defineSeed } from '@pro-laico/payload-seed'

export default defineSeed('posts', ({ ref, asset }) => [
  {
    _key: 'launch',
    title: 'We launched',
    heroImage: asset('post'),
    relatedService: ref('services', 'consulting'), // typed dependency edge across files
  },
])
```

</Tab>

</Tabs>

Records are typed against your app's generated Payload types
(`RequiredDataFromCollectionSlug<slug>`), except relationship and upload fields are widened to also
accept `ref()` / `asset()` tokens. Change a collection's fields and the seed file shows a TS error
until it matches; unknown fields are rejected too. The producing side (`_key`) stays a free string;
only the consuming side (`ref` / `asset`) is checked.

## Assets

`defineAssets(specs)` — declare the uploadable files. Each key becomes an `asset(key)` reference;
its spec points at a file in the assets dir (default `assets/`) plus the upload-doc data. The engine
uploads these **first**, so `asset()` tokens resolve to real ids before any content is created.

```ts
// seed/assets.ts
import { defineAssets } from '@pro-laico/payload-seed'

export default defineAssets({
  serviceA: { file: 'service-a.jpg', alt: 'Consulting' },
  hero: { file: 'hero.jpg', alt: 'Hero', focalX: 78, focalY: 32 },
})
```

Each spec is an `AssetSpec`:

export const assetSpec = [
  { name: 'file', type: 'string', required: true, description: `Filename within the assets dir. The loader tolerates an extension mismatch — a spec naming foo.png picks up foo.jpg if that is what is on disk.` },
  { name: 'collection', type: 'CollectionSlug', description: `Upload collection to create this asset in. Defaults to the plugin's assets.collection.` },
  { name: 'alt', type: 'string', description: `Alt text set on the created upload doc.` },
  { name: 'focalX', type: 'number', description: `Focal point X, a percentage measured from the left.` },
  { name: 'focalY', type: 'number', description: `Focal point Y, a percentage measured from the top.` },
  { name: 'data', type: 'Record<string, unknown>', description: `Extra fields to set on the created upload doc.` },
]

<ConfigOptions items={assetSpec} />

`focalX` / `focalY` set the upload's focal point, so seeded images crop to the right subject the
moment they're served (e.g. by [`@pro-laico/payload-images`](/docs/plugins/payload-images)).

## Globals

Globals seed through a sibling helper, `defineGlobalSeed(slug, build)` — a singleton, so no `_key`.
The builder gets the same tokens, and globals are **updated after** all their dependencies exist:

```ts
// globals/SiteSettings/seed.ts
import { defineGlobalSeed } from '@pro-laico/payload-seed'

export default defineGlobalSeed('site-settings', ({ ref }) => ({
  /* SiteSettings data — ref('services', 'consulting') etc. */
}))
```

## Typed references

Run `payload generate:types` and the plugin injects a `SeedRegistry` into `payload-types.ts`, so
`ref('services', 'consulting')` is checked against your real `_key`s — rename or remove a seeded
item and every stale reference becomes a TS error. Unknown collections and unknown asset keys error
too.

It's injected via Payload's own `typescript.postProcess` hook and rides the same `generate:types`
command you already run, so the augmentation is **global with no import** — exactly like Payload's
generated `GeneratedTypes`. Without codegen, refs fall back to runtime validation — safe either way,
fully safe with it.

## Run order

The CLI `payload seed` and `POST /api/seed` share one engine:

1. **Build the model** from the definitions — asset specs, collection records, globals.
2. **Validate** — every `ref` targets a real collection and resolves to a seeded doc, every `asset`
   resolves to a declared key, no duplicate `_key` within a collection, no unknown top-level fields.
   All issues are collected and thrown at once, naming the collection, `_key`, and field.
3. **Build the dependency graph** — every `ref(collection, key)` is an edge — and **topo-sort** it
   (depth-first). A cycle is a hard error that names the cycle.
4. **Clear** the seeded collections (plus the asset upload collections). Upload collections and
   asset-provider collections clear via `payload.delete` so their hooks fire (e.g. external-asset
   cleanup); plain collections are wiped directly.
5. **Upload assets first**, resolving `asset()` keys to ids, and resolve any provider source files
   to absolute paths.
6. **Create** docs in sorted order, resolving `ref` / `asset` / source tokens to real ids at create
   time.
7. **Update globals** after their dependencies exist.

## Running from code

The CLI, endpoint, and admin button are the usual triggers, but the same engine is exported as
`seed()` for scripts, tests, and migrations — anywhere you already have a `payload` instance. It
builds a Local API `req` if you don't pass one and resolves the plugin options:

```ts
import { seed } from '@pro-laico/payload-seed'

const result = await seed({ payload, options: { definitions: [assets, services, posts] } })
// result.created → { media: 3, services: 2, posts: 1 }  (counts per collection, incl. assets)
// result.order   → ['services:consulting', 'posts:launch', …]  (the topo-sorted create order)
```

`seed()` returns a `SeedResult` — the same `{ created, order }` the `POST /api/seed` response carries
(alongside `success: true`) and the CLI logs on completion. Unlike the CLI and endpoint, `seed()` is
**not** behind the `ENABLE_SEED` guard — the gate lives on the entry points, so a test can drive the
real seed directly. It's still destructive; call it deliberately.

## Revalidation

Next.js caches aggressively, so Payload apps usually wire revalidation hooks — `afterChange` /
`afterDelete` handlers that call `revalidatePath` / `revalidateTag` to push fresh content to the
front end. A seed clears and recreates every seeded collection, so firing those hooks per doc would
be a revalidation storm — and under the CLI there is no request scope, so `revalidatePath` can't run
at all.

So the engine sets `context: { disableRevalidate: true }` on every create, update, and delete it
performs. Revalidate hooks that follow Payload's convention — early-returning when
`context.disableRevalidate` is set — skip automatically during a seed:

```ts
// collections/Posts/hooks/revalidatePost.ts
export const revalidatePost: CollectionAfterChangeHook = ({ doc, context }) => {
  if (!context.disableRevalidate) revalidatePath(`/posts/${doc.slug}`)
  return doc
}
```

The seed does **not** revalidate at the end. After it finishes, revalidate once yourself — a
redeploy, your own revalidation call, or nothing at all when it's a local/dev database you're about
to hit fresh.

## External assets (asset providers)

Some collections store their "file" in an external service rather than a Payload upload — e.g.
[`@pro-laico/payload-mux`](/docs/plugins/payload-mux)'s `mux-video`, whose bytes live in Mux. An
**asset provider** lets such a collection seed **like an image asset**: declared with a source token
and referenced anywhere, all within the normal seed run — no custom script.

Responsibilities split by ownership: the **ingest** (the SDK work) lives in the owning plugin, the
**orchestration** stays in the seed engine. A `SeedAssetProvider` is plain config
(`{ token, collection, sourceDir }`) — the owning plugin returns it without importing the seed
package, and the seed engine consumes it without importing the plugin or its SDK. The two stay fully
decoupled.

### Mux video

Register the provider (the mux plugin exports it) and seed a video with the `video()` source token.
The provider adds the token, resolves its file under `assets.dir/<sourceDir>` first, and hands it to
the `mux-video` collection — whose `beforeValidate` hook uploads it to Mux, waits for the asset to
be ready, and folds in the metadata. Because it waits for `ready`, seeded videos carry full
`playbackOptions` and are **immediately playable — no Mux webhook required**.

```ts
import { muxAssetProvider, muxVideoPlugin } from '@pro-laico/payload-mux'
import { seedPlugin } from '@pro-laico/payload-seed'
import videos from './seed/videos'
import pages from './seed/pages'

plugins: [
  muxVideoPlugin(),
  seedPlugin({
    definitions: [videos, pages],
    assets: { dir: 'seed-assets' },          // video files live under seed-assets/videos/
    assetProviders: [muxAssetProvider()],    // token 'video', collection 'mux-video', dir 'video'
  }),
]
```

```ts
// seed/videos.ts — declared like an image asset
export default defineSeed('mux-video', ({ video }) => [
  { _key: 'intro', title: 'Intro', source: video('intro.mp4', { playbackPolicy: 'public', posterTimestamp: 3 }) },
])

// seed/pages.ts — referenced like any other doc; the engine seeds the video first
export default defineSeed('pages', ({ ref }) => [
  { _key: 'home', title: 'Home', heroVideo: ref('mux-video', 'intro') },
])
```

Run the seed (admin button / `POST /api/seed` / `payload seed`) and each `video()` source is
uploaded to Mux, the `mux-video` doc created with its metadata, then every `ref('mux-video', …)`
resolved. Reseeds clear the provider collection via `payload.delete`, so the mux plugin's
`afterDelete` hook removes the old Mux assets too. For one-off programmatic creation outside seeding,
the mux plugin also exports `ingestMuxVideo()`.
