## HIGH

### 1. Prewarm dedupe permanently blocked by a failed job: where-clause matches hasError jobs the runner will never pick

- **Where:** `src/lib/prewarm/enqueue.ts:31` · concurrency (also flagged by data)
- **Problem:** The enqueue dedupe query treats a permanently failed prewarm job as 'pending', so once a job exhausts retries, every future save of that source is deduped against the dead job and prewarm never runs again for that source.
- **Failure scenario:** Sharp (or storage) is broken during one deploy; the imagesPrewarm job for source X fails all 3 attempts (initial + 2 retries) -> payload-jobs row { taskSlug:'imagesPrewarm', input:{sourceId:'X'}, hasError:true, processing:false, completedAt:null }. After the underlying issue is fixed, every enqueue trigger for X — re-upload/replace, focal-point change, and even the `images:prewarm` CLI sweep — finds that row in the dedupe query, sets dupe=true, and returns without queueing. The runner never picks the row (hasError filter), so X is never prewarmed again until someone manually deletes the job document.
- **Fix:** In packages/payload-images/src/lib/prewarm/enqueue.ts:31, add `{ hasError: { not_equals: true } }` to the `and` array so permanently failed jobs no longer count as pending. Optionally also mirror the runner's waitUntil eligibility (`or: [{ waitUntil: { exists: false } }, { waitUntil: { less_than: now } }]` is NOT wanted here — a future-waitUntil job is still legitimately pending and should dedupe — so the hasError clause alone is the correct fix).
- **Status:** 🟢 **Fixed** — added `{ hasError: { not_equals: true } }` to the dedupe `and` array (mirrors Payload's own runnable-jobs query); regression test in `tests/lib/prewarm/enqueue.spec.ts`.

### 2. Public unsigned transform endpoint has no rate limit, an unbounded concurrency queue, and no cap on distinct persisted variants — CPU/memory/storage exhaustion

- **Where:** `src/lib/transform/limit.ts:16` · security
- **Problem:** An unauthenticated attacker can force unbounded work on /api/img/:id. Each distinct (w,h,fit,q,fmt) combination is a cache miss that runs a Sharp transform and PERSISTS a new file + DB row (overrideAccess:true). Parameters snap to a grid but the grid is huge (~82 widths x ~82 heights x 5 fits x 12 quality buckets x 4 formats ≈ 1.6M distinct persisted variants per single source image), and the FIFO concurrency queue that is supposed to protect the host is itself unbounded.
- **Failure scenario:** An unauthenticated attacker scripts a loop against one public image id (default images collection is read: anyone) enumerating w=50,100,...,4096 x h x q(40..95 step5) x fmt(avif/webp/jpeg/png) x fit. Every request is a distinct cacheKey => a real Sharp encode behind the cpus-1 gate AND a new persisted stored file + generated-images row (overrideAccess:true, getVariantBytes.ts:102-116). Sustained requests (a) monopolize the transform gate so legitimate srcset/image requests queue and stall, and (b) accumulate toward ~1.6M variant files + DB rows for that one source, exhausting object storage, DB, and cloud egress/compute cost. Nothing (auth, rate limit, signature, or per-source variant cap) stops it.
- **Fix:** Add a mitigation on the public endpoint. Best: require a signed URL (HMAC over the params using a server secret) so only URLs the app itself minted (via the srcset/variantUrl builders) are honored — reject unsigned/invalid on /img. Alternatively/additionally restrict allowed widths to a small configured srcset ladder rather than the full 1..maxDimension/step grid, and enforce a per-source variant ceiling in generateVariantBytes' persist() (count existing rows where source == src.id and skip persistence past a cap, still serving the bytes). Add rate limiting on /api/img and bound the concurrency queue depth in limit.ts (reject with 503 past a max), so a burst of misses can't park unbounded work behind the gate.
- **Status:** 🟢 **Partially fixed** — bounded the transform wait queue (`limit.ts`): past `max(64, limit×32)` pending transforms (override `IMAGES_TRANSFORM_MAX_QUEUE`) new work is shed with a `TransformOverloadError` → **503 + Retry-After** instead of parking unbounded work behind the gate; regression test in `tests/lib/transform/limit.spec.ts`. 🔵 **Still open (product decision):** the storage/cost-exhaustion vector — one public source can still accumulate ~1.6M distinct persisted variants. The durable fix is HMAC-signed URLs (breaking: changes every emitted URL, needs a secret) and/or a per-source variant ceiling in `persist()` and/or a width allowlist. Also recommended: rate-limit `/api/img` at the edge. These need Chad's call on the security posture and are left as tracked follow-ups.

- **Where:** `src/plugin.ts:69` · api
- **Problem:** The public `transform.sourceSlug` option only points the endpoints/variant-relationship at another collection but never suppresses or retargets the created `images` collection or its enhancements, and is silently overridden by `extendCollection`.
- **Failure scenario:** `imagesPlugin({ transform: { sourceSlug: 'media' } })` with an existing 'media' upload collection: registration succeeds (validation passes), the admin shows a fully-enhanced 'images' collection, but every URL it emits (virtual src/srcset, admin thumbnails, LQIP) hits /api/img/{imagesId} where the endpoint findByIDs 'media' → 404 for all of them; meanwhile 'media' gets no purge hooks or metadata, so replacing a media file serves stale cached variants indefinitely. Separately, `imagesPlugin({ extendCollection: 'media', transform: { sourceSlug: 'uploads' } })` silently ignores sourceSlug with no warning, unlike the other ignored options.

If both options are kept, (a) when sourceSlug retargets without extendCollection, skip registering the default 'images' collection (or apply imageEnhancements to the target instead), and (b) add transform.sourceSlug to the ignoredWithExtend warning list at plugin.ts:191-193. Update tests/plugin.spec.ts:85 and configuration.mdx:24 to match the chosen behavior.
- **Status:** 🔴

### 4. Stale variant resurrected after purge: in-flight request's deferred persist lands post-purge and the fallback picker then serves the old-focal/old-file bytes

- **Where:** `src/lib/transform/getVariantBytes.ts:127` · concurrency
- **Problem:** A transform request (or prewarm job) that read the source doc before an identity change persists its variant after purgeStaleVariantsAfterChange deleted everything, re-inserting a stale row that the fallback picker will serve as a stand-in — for focal-only edits the stale variant is a perfect geometric match, so visitors see the old crop.
- **Failure scenario:** t0: GET /img/42?w=1200&ar=16:9 misses the exact key; Sharp starts with focal (30,40). t0+50ms: editor saves focal (70,60); afterChange purge deletes all of source 42's variants inside the update transaction. t0+300ms: the first request's response flushes and after(persist) inserts the (30,40)-crop variant row (insert succeeds: source alive, unique cacheKey). t0+2s: GET /img/42?w=800&ar=16:9 — exact key (new focal) misses; the fallback query (source equals 42) finds the resurrected row; pickFallbackVariant matches it on fit+geometry (focal doesn't change dimensions) with no focal comparison, so the OLD-crop bytes are served as the stand-in. The stale row also lingers as an eligible fallback candidate for every other not-yet-generated shape until the next identity change purges it. Same race applies to a prewarm job that read the source pre-edit and persists post-purge (deferPersist:false, no after(), but the transform duration is still a window).
- **Fix:** In `persist` (getVariantBytes.ts:100), before payload.create, re-read the source doc (findByID, depth 0, select the identity fields) and recompute variantCacheKey from the fresh doc; if it no longer equals `key`, skip the create and log at info (mirroring the existing "source was deleted before variant persisted" branch). Belt-and-braces: have the fallback path select focalX/focalY (endpoints/transform/index.ts:100-110) and have pickFallbackVariant reject candidates whose stored focalX/focalY differ from the current source doc's — this also shields against rows persisted by other processes/jobs that raced the same edit.
- **Status:** 🔴

### 6. detectVariantIdentityChange misses same-filename file replacement — stale variants and stale version token served forever

- **Where:** `src/hooks/collection/variantIdentity.ts:17` · data
- **Problem:** File replacement is detected solely by `previousDoc.filename !== doc.filename`, but Payload keeps the filename identical whenever overwriteExistingFiles is true, so replaced bytes never purge variants or bust caches.
- **Failure scenario:** Two concrete paths: (1) `payload.update({ collection: 'images', id, filePath: './hero.jpg', overwriteExistingFiles: true })` with new bytes — filename/focal/hotspot unchanged → afterChange hooks see no identity change → variants not purged, prewarm not enqueued, variantCacheKey and v= token identical → /api/img keeps serving the OLD pixels from generated-images, and browsers/CDNs keep the old bytes under the immutable header indefinitely. (2) Editor uses Payload's built-in crop tool in the admin (enabled by default) without moving the focal point — uploadEdits.crop forces overwriteExistingFiles=true, the cropped bytes overwrite the same filename → same silent staleness. Only the doc's original-file endpoint shows the new image; every transform URL stays stale forever.
- **Fix:** Fold byte-sensitive signals into the shared identity in lockstep across all three hash sites: add filesize (plus width/height) to detectVariantIdentityChange's fileChanged check, to variantCacheKey's parts, and to deriveVersion's hash input (they're already on the upload doc, cost nothing, and change on any crop/resize and virtually any byte replacement). Alternatively store a short content hash in a beforeChange hook when req.file is present and key on that for exhaustive coverage. Also consider setting upload.crop:false on the Images collection (the plugin has its own hotspot model) and documenting that overwriteExistingFiles-style replacement otherwise requires a manual purge.
- **Status:** 🔴

### 8. Fallback rule B accepts any candidate whose ratio lands in the snap quantization band — a 16:9 crop can be served for a 3:2 request and even outrank the correct-family candidate

- **Where:** `src/lib/transform/fallback.ts:46` · geometry
- **Problem:** Rule B (`replayHeight(p.w, c.width / c.height) === p.h`) treats every candidate whose aspect ratio falls anywhere inside the request height's snap bucket (±half a dimensionStep, i.e. ±25px on a 50px grid) as an 'identical crop', so distinctly different crop families qualify; scoring then prefers the smallest sufficient width, letting the wrong-ratio candidate beat a correct one.
- **Failure scenario:** Source 3000×2000. Cache holds a 16:9 variant 800×450 (from ?w=800&ar=16:9) and a true 3:2 width-only variant 800×533. Request ?w=400&ar=3:2 (snapped to 400×250) misses the cache → pickFallbackVariant qualifies BOTH via rule B (replay(400, 800/450)→h=225→snaps to 250; replay(400, 800/533)→h=267→snaps to 250); scoring ties and the DB-order-first doc wins, so the 16:9 crop can be served into the 3:2 slot. Worse, deterministic variant: with a 7:4 variant 700×400 in the pool, it qualifies and strictly beats the correct 800×533 (700 < 800, both ≥ effectiveW=400) — a ~17% wrong-ratio crop is painted (squashed or CSS-cropped) even though the correct-crop stand-in exists. Mirror defect in rule A confirmed: request 800×450 accepts a 400×250 (ratio-1.6) candidate. All reproduced by running the real code.
- **Fix:** In packages/payload-images/src/lib/transform/fallback.ts: (1) add ratio closeness as the LEADING score key — e.g. prepend Math.abs(c.width/c.height - reqRatio) (or a small-epsilon bucket of it) to the score tuple at line 55 — so a correct-family candidate always beats a band-straggler; this alone fixes the deterministic wrong-pick. (2) Tighten qualification: after a rule A/B replay match, additionally require the candidate's true ratio to be within the snap band at the CANDIDATE's own width (where the band is proportionally narrower), i.e. Math.abs(c.height - c.width / reqRatio) <= constraints.dimensionStep / 2 + 1 — this keeps the legitimate snap-distortion matches the existing tests assert (650×350↔1600×900: |350-365.6|=15.6 ≤ 26) while rejecting genuine cross-family candidates at widths where half a step no longer spans families (800×450 vs 1.6: |450-500|=50 > 26; 700×400 vs 1.6: |400-437.5|=37.5 > 26). Do NOT require rules A and B jointly — the existing legitimate case at fallback.spec.ts:34 passes only rule B. Add regression tests: 400×250 request must reject 800×450 and 700×400 and must pick 800×533; 800×450 request must reject 400×250.
- **Status:** 🔴

### 9. Fallback cannot distinguish hotspot-zoomed (w+h) variants from full-frame (width-only) variants with identical dims — wrong content served when focalSize < 100

- **Where:** `src/lib/transform/fallback.ts:47` · geometry
- **Problem:** Width-only cover renders skip hotspotWindow entirely (no zoom, no focal centering) while two-dimension renders apply focalSize zoom, yet both persist with fit='cover' and only width/height/fit/format/quality are visible to the picker — so when the source ratio matches the requested ratio their dims coincide exactly and rule C (±1px) swaps one for the other.
- **Failure scenario:** Source 1600×900 with focalSize=50 (2× hotspot zoom), default focal 50/50. A plain `<img src="/api/img/:id?w=800">` (exactly what buildVariantUrl emits with no aspectRatio) renders via the sharp.ts else branch → stores a FULL-FRAME 800×450 row with fit='cover'. A later request `?w=800&h=450` (or `?w=800&ar=16:9`) hashes to a different cacheKey (h is in the key) → cache miss → pickFallbackVariant finds the 800×450 candidate, fit matches, replayHeight(800, 16/9)=450 → qualifies → the visitor is served the un-zoomed full frame in place of the author's 2× tight hotspot crop (Cache-Control: no-store, corrected on the next request once the background generate lands). The reverse swap — zoomed 800×450 standing in for a full-frame width-only request — occurs symmetrically via the source-ratio anchor at fallback.ts:35.
- **Fix:** Persist which render path produced the variant and require it to match in the picker. Concretely: in generateVariantBytes persist() (getVariantBytes.ts:105-113) store `windowed: p.h != null` (a checkbox/boolean field added to the generatedImages collection); add `windowed?: boolean | null` to FallbackCandidate and to the endpoint's candidate select (endpoints/transform/index.ts:100-110); in qualifies() (fallback.ts:40), when the source's hotspot actually diverges the two paths — focalSize non-null and < 100 (the picker already receives the source doc; widen its param type to include focalSize) — reject candidates whose `windowed` (treat null/legacy rows as unknown → reject) differs from `p.h != null`. When focalSize is 100/unset the renders are identical and the current matching stays as-is, preserving all existing fallback hits.
- **Status:** 🔴

### 10. maxDimension cap on an ar-derived (or explicit) height silently destroys the requested aspect ratio instead of scaling both dimensions

- **Where:** `src/lib/transform/params.ts:92` · geometry
- **Problem:** When the derived (or supplied) second dimension exceeds maxDimension it is clamped independently (`cap` at params.ts:87, applied at :88-89 and :92-93), so the served cover crop has a completely different aspect ratio than requested — and different srcset rungs of the same image get different crops.
- **Failure scenario:** Source 4000×6000, buildSrcset with aspectRatio '9:16' (fit=cover default). stepWidths yields rungs 50..3950 + 4000. Rungs w≤2304 request h≤4096 and render true 9:16 hotspot crops; rung w=3000 requests h=5333 which parseTransformParams caps to 4096, so sharp.ts crops at ratio 3000/4096=0.732; rung w=4000 serves a 4000×4096 (ratio 0.977) near-square crop. A 2x-DPR phone picking a high rung shows a visibly different cropping (much more horizontal context, subject smaller) than a 1x laptop picking a low rung, and the tall rungs letterbox/distort inside the 9:16-styled layout box.
- **Fix:** In parseTransformParams (packages/payload-images/src/lib/transform/params.ts), when a cap actually clips a dimension and an aspect ratio is in play (explicit `ar`, or both w and h supplied), rescale the partner dimension to preserve the ratio before the grid snap: e.g. derive h uncapped; if h > c.maxDimension, set h = c.maxDimension and w = cap(Math.round(h * ar)) (symmetric when w is the clipped one; for the both-dims-supplied case use ar = wRaw/hRaw). Optionally mirror the same clamp in buildVariantUrl/stepWidths so the emitted srcset never asks for a dimension the endpoint will refuse — keeping URL `w` and served width consistent (the `${w}w` descriptor would otherwise lie once the server shrinks w).
- **Status:** 🔴

### 11. Original-image Buffers are loaded before (outside) the transform concurrency gate, once per variant, with an unbounded wait queue

- **Where:** `src/lib/transform/getVariantBytes.ts:67` · perf
- **Problem:** On a cache miss, generate() reads the FULL original into a Buffer (up to MAX_FETCH_BYTES = 64MB) before calling transformImage, but only transformImage is gated by withTransformLimit — so under a concurrent-miss burst every queued request holds a whole-original Buffer in memory while waiting for a Sharp slot, and the limit.ts queue itself is unbounded.
- **Failure scenario:** Cold cache (fresh deploy, or purge after a focal-point edit via purgeStaleVariantsAfterChange) on a memory-constrained Node host with IMAGES_TRANSFORM_CONCURRENCY / cpu-derived limit = 1-2. A page fires ~20 concurrent transform misses over 30MB originals (each under the 64MB fetch cap). All 20 `generate()` calls immediately complete `readBytes` — ungated — so ~600MB of original Buffers sit resident while 18-19 requests wait in limit.ts's uncapped queue for a Sharp slot; RSS spikes past a 1GB function limit → OOM kill, and every in-flight image request 5xx/aborts. Per-source variant: 12 concurrent width/format misses of one 40MB original allocate 12 separate copies (~480MB) because coalescing is per-variant-cacheKey and sourceFlight dedupes only the doc lookup, never the bytes.
- **Fix:** In generateVariantBytes (packages/payload-images/src/lib/transform/getVariantBytes.ts), move the `readBytes` call inside the gated region — e.g. wrap the read+transform in `withTransformLimit` (exporting it for use here, and having transformImage take an already-acquired path or a `gated: false` variant so the slot isn't acquired twice), so at most `limit` original Buffers are resident. Add a per-source single-flight (keyed on `${src.id}|${src.filename}`) around the original-bytes read so concurrent variants of one source share a single Buffer while any of them is in flight. Optionally cap limit.ts's queue length and reject overflow with 503 + Retry-After to shed thundering-herd load instead of queueing unboundedly.
- **Status:** 🔴

### 12. prewarmSource re-reads/re-fetches the entire original once per target (default 24 times per job)

- **Problem:** The prewarm loop calls getOrCreateVariantBytes per target, and every miss re-runs generate() → readBytes(src, …), so one prewarm job reads or remote-fetches the full original up to maxVariantsPerImage (default 24, resolveOptions.ts:16) times instead of once.

### 13. Default 50px pixelStep makes srcset emit ~82 URLs (~8KB) per doc on every read, and multiplies the server-side variant space past the fallback query's page size

### 14. Fallback candidate query fetches an arbitrary unsorted first-100 variants per cache miss — misses valid stand-ins once a source exceeds 100 variants, and pays the query on every miss

- **Problem:** The nearby-fallback lookup does `find({ where: { source: equals }, limit: 100 })` with no sort and no narrowing (fit/format/width), then filters the 100 rows in JS — so a source with more than 100 variants gets an arbitrary page (Payload default createdAt order) in which a qualifying same-geometry stand-in may simply not appear, while every cache miss (including brand-new sources with zero variants) pays this extra 100-doc query plus up to ~200 parseTransformParams replays.

Optionally skip the query entirely for sources with no variants via a cheap count or by keying off a recent-miss marker. Keep pickFallbackVariant unchanged as the final arbiter.

### 33. PNG variants persist the request's quality label even though the cache key forces q=0 — the stored quality is a lie and skews fallback tie-breaking

- **Where:** `src/lib/transform/variantKey.ts:12` · geometry
- **Problem:** variantCacheKey correctly collapses all png qualities to one key (encode() ignores quality for png), but generateVariantBytes persists `quality: p.q` from whichever request happened to generate the variant first, so the row's quality field is arbitrary (40..95) and pickFallbackVariant's quality-distance tie-break (fallback.ts:58) scores png candidates against a meaningless number.
- **Failure scenario:** Source image (id S) with transparency. Request A: /img/S?w=1000&fmt=png&q=40 → variant row persisted with quality=40 (key collapsed q to 0). Request B: /img/S?w=1000&fmt=jpeg&q=90 → jpeg row with quality=90. Request C: /img/S?w=800&fmt=png&q=90 → cache miss (different width); fallback pool contains both w=1000 candidates, both qualify and tie on tiers 1-2 (same width ≥ 800). Tier 3: png |40-90|=50 vs jpeg |90-90|=0 → the jpeg wins and is served as the stand-in, flattened to a white background, while an identical-crop png (whose bytes are what q=90 would have produced anyway) sits unused. Had the png row stored null (or the request's q), tier 3 would tie at 0 and tier 4 (format match) would correctly pick the png.
- **Fix:** In generateVariantBytes (packages/payload-images/src/lib/transform/getVariantBytes.ts:110) persist the same normalized value the key uses: `quality: format === 'png' ? null : p.q` — null makes fallback.ts:58's `c.quality ?? p.q` score png candidates at distance 0, letting the format tie-break decide. Optionally also skip the quality tie-break for png candidates explicitly in pickFallbackVariant (fallback.ts:58: treat `c.format === 'png'` as distance 0) to be robust against rows persisted before the fix. Add a test asserting a png candidate with a stale quality label is not deprioritized against a same-width lossy candidate.
- **Status:** 🔴

### 34. SSRF host guard misses IPv4-mapped IPv6 and does no DNS resolution (rebinding / hostname->private IP)

- **Where:** `src/lib/transform/source.ts:14` · security
- **Problem:** isPrivateHost only pattern-matches literal dotted-IPv4 and fe80/fc-fd IPv6 hostnames. It does not catch IPv4-mapped IPv6 literals, and the guard runs on the URL hostname string only — never on the resolved IP — so any hostname that DNS-resolves to a private/link-local address passes. Cloud metadata (169.254.169.254 / metadata.google.internal) and internal hosts are reachable when a source doc's url is attacker-influenced.
- **Failure scenario:** Concrete guard-level failure: call isAllowedFetchTarget for a target whose hostname is ::ffff:a9fe:a9fe (i.e. doc.url = 'http://[::ffff:169.254.169.254]/latest/meta-data/iam/security-credentials/'). isPrivateHost returns false, so readBytes proceeds to fetch() and, because ::ffff:169.254.169.254 routes to the mapped IPv4 at the OS level, connects to the cloud metadata service, pulling internal content into the transform pipeline. Same for any hostname that DNS-resolves to a private/link-local address (no resolution check). This requires the source-doc url host to be attacker-set, which normal Payload virtual-url semantics do not permit — so it is a real bypass of the stated control but not independently exploitable through the shipped code paths I traced.
- **Fix:** In isPrivateHost, after stripping brackets, normalize and reject IPv6-embedded IPv4 before the plain-IPv4 regex: match /^::ffff:(\d{1,3}\.){3}\d{1,3}$/i and /^::(\d{1,3}\.){3}\d{1,3}$/i, and also handle the hex-quibble form (::ffff:a9fe:a9fe) by expanding the last 32 bits to dotted IPv4 and running it through the existing IPv4 checks; additionally reject ::1 (already), and IPv4-in-IPv6 compat. For rebinding/DNS coverage, resolve the hostname via dns.lookup({all:true}) and run every returned address through the private/reserved-range check before fetching, ideally pinning the fetch to the vetted IP (or restrict egress to an allowlist of CDN hosts). Add these forms to the SSRF test in source.spec.ts.
- **Status:** 🔴
